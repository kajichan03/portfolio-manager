# Project Product Design Skill

> 产品需求梳理和原型设计方法论

---

## 概述

本 Skill 提供产品从需求梳理到原型设计的完整工作流，适用于：
- 新产品功能规划
- 产品需求文档（PRD）编写
- 交互原型设计与验证

---

## 文档体系

```
projects/{name}/
├── Project_Goal.md          # 项目章程（需求背景+目标）- 用户维护，AI 只读
├── progress.md              # 阶段进度仪表盘 - 每轮对话更新
├── decisions.md             # 关键决策记录
├── input_product_doc.md     # 现有产品文档（参考输入）
└── output_product_doc.md    # 更新后的产品文档（最终产出）
```

> **⚠️ 重要**: Project_Goal.md 由用户手工维护，AI Agent **严禁修改**。

---

## 七阶段工作流

### 阶段 1：明确项目背景和目标

**用户动作**：编写 Project_Goal.md 的项目背景、目标用户、核心目标

**AI 动作**：
- 阅读 Project_Goal.md 理解项目
- 在 progress.md 记录阶段完成

---

### 阶段 2：拆解需求和功能点

**用户动作**：在 Project_Goal.md 中细化功能需求

**AI 动作**：
- 基于 Project_Goal.md 生成两个任务列表：
  - **需求文档任务列表**：以功能点为单位的 PRD 编写任务
  - **原型页面设计任务列表**：以页面为单位的设计任务
- 在 progress.md 记录里程碑完成

---

### 阶段 3：编写需求文档

**AI 动作**：
- 按任务列表推进 PRD 编写
- 每个功能点需包含：用户故事、业务规则、验收标准
- 在 progress.md 更新任务进度

---

### 阶段 4：验证需求结构

**AI 动作**：
- 生成「整体信息架构总览」检查结构合理性
- 确认功能模块划分是否清晰
- 在 progress.md 记录验证完成

---

### 阶段 5：推进原型设计

**设计任务分类**：

| 任务类型 | Prompt 来源 | 说明 |
|----------|-------------|------|
| **全新设计任务** | 调用 `prototype-prompt-generator` 生成 Prompt | 从未设计过的新功能 |
| **需求变更导致的重新设计** | **直接使用需求变更文档（decisions.md）作为 Prompt** | 已有设计但因需求变更需调整 |

**步骤（强制门禁）**：

1. **触发识别（先做）**：
   - 当用户请求包含以下意图时，必须进入本流程：
     - 标记完成 / 更新进度 / 改状态
     - Dxx 或设计任务状态变更

2. **强制前置清单（先校验，后写入）**：
   - `skill命中`：已命中 `project-management-product-design`
   - `任务类型`：确认是设计任务（Dxx）还是需求任务（Rxx）
   - `交付物证据`：是否存在设计方案链接（Figma/设计稿 URL）
   - `状态变更许可`：通过 / 不通过
   - `缺失项`：如“缺 D16 设计链接”

3. **判断任务类型**：
   - 全新设计 → 调用 `prototype-prompt-generator` 生成 Prompt
   - 需求变更重新设计 → **直接使用 decisions.md 中的变更文档作为 Prompt**

4. 用户基于 Prompt 完成设计后，**必须提供设计方案链接**

5. **两阶段提交（禁止跳步）**：
   - 阶段 1：仅校验，不改文档状态
   - 阶段 2：仅当“状态变更许可=通过”时，更新 progress.md（状态 + 交付物链接 + 统计）

6. **【检查点】收到交付物后，才能标记任务为"已完成"**

7. 更新 progress.md（状态 + 交付物链接）

**交付物检查规则（强制）**：

| 状态变更 | 交付物要求 | 操作 |
|----------|-----------|------|
| 🚧 待设计 → 🔄 设计中 | 无 | 允许 |
| 🔄 设计中 → ✅ 已完成 | **必须有设计方案链接** | 无交付物则保持"设计中" |
| 任何状态 → ❌ 已取消 | 无 | 允许 |

**硬规则补充**：
- 设计任务（Dxx）改为 `✅ 已完成` 时，必须有设计方案链接。
- 无链接时，仅允许：
  - 保持原状态；或
  - 标记为 `🚧 待补交付物（缺链接）`。
- **禁止**“先标记完成、后补交付物”。

**执行输出模板（每次状态变更前必须给出）**：
- `命中 skill`：是/否
- `交付物检查`：通过/未通过
- `允许标记完成`：是/否
- `缺失项`：xxx

**示例**：
- D9 因 DEC-008 需求变更需重新设计 → 直接使用 DEC-008 文档作为设计输入，无需重新生成 Prompt

---

### 阶段 6：汇总产品文档

Prompt：

```
作为产品经理，我希望基于 Project_Goal.md 中的项目目标和需求内容，按照 input_product_doc.md 中的格式，生成更新版的产品文档。目标是产出一个具体、清晰的产品规则文档。如果有任何不确定的地方，请标记并向我提问以进行确认。
目标：
1. 每个功能有具体、清晰的产品规则
2. 每个功能记录设计原因（用户故事）
3. 不要记录项目管理的相关信息（比如任务编号）
```

**AI 动作**：
- 使用上述的固定的 Prompt。
- 在 progress.md 记录里程碑完成
- **【关键文档待 Review】** 提示用户 Review，记录反馈

---

### 阶段 7：需求变更处理

**触发**：需求发生变更时

**AI 动作**：
1. 更新 decisions.md 记录变更原因
2. 将相关任务状态改为"需求变更，待更新"
3. 提示更新产品文档
4. 给出原型修改建议（增量更新而非重新生成）

---

## 每日协作流程

### 对话开始时
```
1. 阅读 progress.md 了解当前阶段和进度
2. 阅读 Project_Goal.md 了解项目章程（只读）
3. 阅读 decisions.md 了解历史决策
```

### 对话结束时
```
1. 更新 progress.md 记录当前阶段进度
2. 如有新决策，更新 decisions.md
3. 如有待确认内容，明确提示用户 Review
```

## 关键机制

### 文档更新触发条件

| 场景 | 更新文档 | 内容 |
|------|---------|------|
| 阶段推进 | progress.md | 阶段状态、任务进度 |
| 需求/设计决策 | decisions.md | 决策内容、原因、影响 |
| 完成 PRD | output_product_doc.md | 完整产品文档 |

### 交付物强制检查

标记任务为"✅ 已完成"时，**必须**确认：
- [ ] 设计任务：有 Figma/设计稿链接
- [ ] 文档任务：内容已写入 output_product_doc.md

**禁止**：交付物为"待补充"或空值时标记完成。

### 状态变更两阶段提交

对所有任务状态变更（尤其是 Dxx）执行：
1. 阶段 1（校验）：输出“强制前置清单”结果，不写入文档。
2. 阶段 2（提交）：仅在校验通过时写入 progress.md，并同步统计。

---

## 最佳实践

1. **用户维护章程**：Project_Goal.md 由用户编写，AI 只读取
2. **双任务列表**：需求任务和原型任务并行跟踪
3. **设计交付物**：每个设计任务必须有设计方案链接
4. **变更影响追踪**：需求变更需同步更新所有相关任务状态
5. **小步快跑**：单个设计任务控制在 1-2 天内完成
