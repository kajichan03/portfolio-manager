### 2026-03-10 - 目录结构重组 - 分离基础设施与项目产出

**日期**: 2026-03-10

**背景**: 
根目录混杂基础设施（AGENTS.md）、过程文档（requirements.md 等）和项目产出（projects/ 等），边界模糊；项目完成后需能独立交付产出，不应包含过程文档。

**决策**: 
创建 `output/` 目录，将 `projects/`、`tasks/`、`resources/`、`metrics/` 移入其中。

**原因**: 
- 保证项目管理 skill 和 cli agent 正常工作（基础设施和过程文档保持原位）
- 项目完成后直接复制 `output/` 即可交付，无需手动筛选
- 符合 Skill 要求："项目产出独立可迁移"

**影响**: 
- `README.md`、`AGENTS.md` 中的路径已同步更新
- 所有后续引用这些目录的操作需使用新路径

### 2026-03-10 - 任务粒度规则：Reminders 列表即项目

**日期**: 2026-03-10

**背景**: 
需要明确 Apple Reminders 与 Portfolio 之间的映射关系，避免任务粒度混乱。

**决策**: 
1. **项目定义**：Reminders 的一个任务列表 = Portfolio 中的一个项目
2. **任务定义**：列表内的具体事项 = 项目中的任务
3. **同步策略**：同步所有任务列表（即所有项目）
4. **立项方式**：新建 Reminders 列表 = 新建项目，自动进入 Portfolio
5. **结项方式**：项目完成后，归档 Portfolio 数据 + 删除 Reminders 列表

**原因**: 
- 约定优于配置，减少手动标记的心智负担
- 符合用户现有工作流（已优化 Reminders 结构）
- 项目边界清晰（列表即边界）

**影响**: 
- Portfolio 需要支持动态项目发现（列表增删）
- 需要设计项目归档机制

### 2026-03-10 - 产品形态决策：Skill + CLI 混合架构

**日期**: 2026-03-10

**背景**: 
需要确定 Portfolio 的实现形态，满足对话交互和定时报告需求。

**决策**: 
采用 **Skill + CLI 混合架构**

| 组件 | 职责 | 实现方式 |
|------|------|----------|
| **Portfolio Skill** | 对话交互、任务分配、状态查询、配置定时任务 | OpenClaw Skill (`SKILL.md` + scripts) |
| **Portfolio CLI** | 数据操作、定时报告生成 | Python 脚本 |
| **系统 Cron** | 定时触发报告 | `crontab` 或 OpenClaw cron |

**交互流程**:
```
用户对话 → Clawd(Skill) → 调用 CLI 脚本 → 操作数据 → 返回结果
                  ↓
           配置定时任务 → Cron 定时调用 CLI → 发送报告
```

**Skill 边界**:
- ✅ 解析自然语言指令（"状态"、"分配 xxx 给 pd"）
- ✅ 协调其他 Agent（通知 pd/Dev 开始工作）
- ✅ 配置定时报告（设置时间、渠道）
- ✅ 调用 CLI 生成 Dashboard

**CLI 边界**:
- ✅ 读取 Reminders API
- ✅ 解析 progress.md
- ✅ 生成文本 Dashboard
- ✅ 发送 Telegram/iMessage 报告

**原因**: 
- Skill 提供必要的自然语言理解和 Agent 协调能力
- CLI 独立处理数据操作，支持定时触发
- 职责分离，Skill 专注交互，CLI 专注数据处理

**影响**: 
- 需要开发两部分：Skill 目录 + CLI 脚本
- 数据路径：Skill 通过配置或约定访问 `output/` 数据

### 2026-03-10 - P1 需求范围确定（简化设计）

**日期**: 2026-03-10

**背景**: 
需要确定 P1 级别需求的范围，避免过度设计。

**决策**: 

| 需求 | 决策 | 原因 |
|------|------|------|
| **项目范围** | 显示所有项目 | 管理生活中所有事情，不限制类型 |
| **项目类型** | 两种类型 | ① 专门项目（有开始有结束的大事情）<br>② 日常运营项目（某个主题下的琐事） |
| **项目依赖关系** | ❌ 不需要 | 避免过度设计，一般无依赖 |
| **时间工时追踪** | ❌ 不需要 | 避免过度设计 |
| **度量指标** | ❌ 不过度设计 | MVP 阶段不定义复杂指标 |
| **实时状态查看** | 询问主 Agent → Dashboard 输出 | 简单直接，主动查询 |
| **定时报告** | 定时任务 → Telegram + iMessage | 被动接收，定时推送 |

**原因**: 
- 用户目标：管理生活中所有事情，范围不限
- 设计原则：Done > Perfect，避免过度设计
- 交互模式：主 Agent 作为统一入口，Dashboard 作为信息载体

**影响**: 
- 数据模型需支持"项目类型"字段（专门项目 vs 日常运营）
- 需要设计 Telegram 和 iMessage 的定时推送机制
- 不实现：依赖图、工时统计、复杂度量指标

### 2026-03-10 - 优先级字段 scheme

**日期**: 2026-03-10

**背景**: 
需要确定优先级标记方式，用于 Dashboard 排序（不控制可见性）。

**决策**: 

**1. 优先级来源**
- Apple Reminders：使用系统内置的优先级字段（非自定义标签）
- Portfolio：只读同步，不修改源数据

**2. 优先级档位**
| 档位 | Reminders 值 | 说明 |
|------|--------------|------|
| 高 | `high` | 优先处理 |
| 中 | `medium` | 默认档位 |
| 低 | `low` | 延后处理 |

**3. Bot 项目同步**
- progress.md 中使用相同三档制：`high` / `medium` / `low`

**4. Dashboard 展示规则**
- 排序：high → medium → low
- 显示：所有项目（优先级不控制可见性）
- 用途：优先级用于需求分析阶段确定范围，而非过滤显示

**原因**: 
- 与 Reminders 系统字段保持一致，无需额外标记
- 三档制简单明确，满足需求范围确认需要
- 所有项目可见，避免遗漏低优先级任务

**影响**: 
- Dashboard 实现时需支持三级优先级排序
- Bot 项目需要按此 scheme 更新 progress.md 模板

### 2026-03-10 - Projects 列表维护与 Dashboard 查看需求

**日期**: 2026-03-10

**背景**: 
明确 Projects 列表的维护方式和 Dashboard 的查看需求。

**决策**: 

**1. Projects 列表维护**
| 属性 | 定义 |
|------|------|
| 维护者 | Clawd（主 Agent） |
| 项目地址 | 登记时手动指定 |
| 存储位置 | `output/projects/projects.yaml` |

**2. Dashboard 查看方式**
| 属性 | 定义 |
|------|------|
| 查看频率 | 主动打开查看（非定时推送） |
| 触发场景 | 实时状态查询、任务分配前、Review 前 |

**3. Dashboard 必须展示内容**
- [x] 所有项目列表及进度百分比
- [x] 每个项目的当前阶段/任务，有哪些 Agent 正在推进
- [x] 阻塞/等待 Review 的任务
- [x] 即将到期的任务

**不展示**（本次范围外）：
- [ ] 各 Agent 的工作负载统计

**原因**: 
- 主 Agent 模式：Clawd 统筹项目，执行 Agents 只负责推进
- 手动指定地址灵活，适应不同项目结构
- 主动查看符合"实时查看"需求，避免推送干扰

**影响**: 
- Dashboard 需要支持实时刷新
- 需要定义"即将到期"的时间阈值

### 2026-03-10 - Bot 与项目数据模型设计

**日期**: 2026-03-10

**背景**: 
需要设计 Portfolio 如何追踪 bots 和 bot 项目，支持资源可视化和任务分发。

**决策**: 
采用 **Agent-项目分离** 模型，维护两个独立列表：

**1. Bots 列表**（Agent 元数据）
| 字段 | 说明 |
|------|------|
| 名称 | Agent 唯一标识 |
| Emoji | 可视化标识 |
| 职责类型 | 该 Agent 的能力/角色 |
| 通信渠道 | 如何与该 Agent 交互 |
| 配置位置 | Agent 配置文件的本地路径 |

**2. Projects 列表**（项目元数据）
| 字段 | 说明 |
|------|------|
| 项目名称 | 唯一标识 |
| 描述 | 项目简介 |
| 本地地址 | 项目工作目录（Agent 可直接使用） |
| GitHub 地址 | 仓库链接 |

**3. 关联关系**
- 支持多对多：一个项目可有多个 Agent，一个 Agent 可参与多个项目
- 仪表盘展示：每个项目的 Agent 参与情况 + 进度

**4. 部署假设**
- 当前：所有 Agent 运行在同一台电脑，共享本地地址
- 未来：可扩展支持多机（增加"电脑名称"字段）

**原因**: 
- 解耦 Agent 能力与项目地址，灵活分配
- 分发任务时可直接提供项目本地地址，Agent 无需查找
- 支持同一项目多 Agent 协作，各负责不同部分

**影响**: 
- 需要设计两个列表的维护方式（手动维护 vs 自动发现）
- 需要设计项目-Agent 分配机制
